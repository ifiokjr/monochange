# Design Note: GitHub Release Automation, Changelog Formats, Release PRs, Deployments, and Changeset Bot Rules

## Status

Planning draft prepared in isolated worktree for discussion.

- Worktree: `/Users/ifiokjr/Developer/projects/monochange-github-automation`
- Branch: `spike/github-release-automation-plan`

## Goal

Extend MonoChange from local planning and workflow-driven file mutation into a GitHub-aware release automation system that can:

1. render changelogs in configurable formats
2. publish GitHub releases from prepared release targets
3. open and maintain release or changeset pull requests
4. trigger automatic deployments when release pull requests merge
5. enforce pull-request changeset rules through a GitHub bot or bot-like workflow

## Current State

MonoChange already supports:

- top-level workflow-defined commands
- `mc validate`, `mc discover`, `mc change`, and `mc release`
- grouped and ungrouped release targets
- package and group changelogs
- release target metadata for `tag`, `release`, and `version_format`
- generic `Command` workflow steps with dry-run support

MonoChange does **not** yet provide first-class support for:

- GitHub release publication
- release-note templating beyond current changelog file writes
- opening or updating GitHub pull requests
- merge-driven deployment orchestration
- PR-time bot enforcement for missing or invalid changesets

## Product Principles

### 1. Keep MonoChange config-driven

Repositories should describe automation in `monochange.toml`, not scatter release semantics across ad hoc GitHub workflows.

### 2. Separate planning from execution

MonoChange should keep its strong planning core and make side effects explicit:

- planning computes what should happen
- GitHub release publication publishes that plan
- PR creation proposes that plan
- deployment executes post-merge automation

### 3. Keep repository-specific deploys customizable

MonoChange should understand _when_ deployments should run and _what release metadata exists_, but it should avoid baking every deployment platform into core.

### 4. Prefer first-class release domain models over shell-only automation

The existing `Command` step is useful, but GitHub releases, release PRs, and changeset policies are rich enough to deserve typed configuration and typed runtime behavior.

### 5. Start GitHub enforcement as workflow automation, then evolve into a GitHub App if needed

A GitHub Action-based bot can validate the product and config model quickly. A dedicated GitHub App can follow if richer review, commenting, and merge orchestration are needed.

## Recommended Capability Breakdown

## Capability A — Changelog formats

### Why

Today MonoChange can write changelog files, but it does not have an explicit changelog _format_ model. GitHub releases and release PRs need structured notes, not only raw file updates.

### Recommendation

Introduce a typed changelog rendering layer with reusable release-note sections.

### Proposed formats

- `keep_a_changelog`
- `cargo_like`
- `changesets_like`
- `markdown_sections`
- `custom_template`

### Proposed config direction

```toml
[defaults.changelog]
format = "keep_a_changelog"
include_links = true

[package.monochange.changelog]
format = "cargo_like"

[group.main.changelog]
format = "markdown_sections"
sections = ["summary", "packages", "evidence"]
```

### Design implications

- changelog rendering should consume a structured release-note model
- GitHub release notes and PR bodies should reuse the same rendered sections
- package and group changelog output should continue to support overrides

## Capability B — GitHub releases

### Why

MonoChange already computes release targets with tag names, versions, and group membership. That is the right substrate for GitHub release publication.

### Recommendation

Add first-class GitHub release publication support based on prepared release targets.

### Proposed runtime behavior

For each release target where `release = true`:

- ensure the tag exists or create it
- create or update the GitHub release
- attach rendered release notes
- optionally mark prerelease or draft
- optionally upload release artifacts generated by repository workflows

### Proposed config direction

```toml
[github]
owner = "ifiokjr"
repo = "monochange"

[github.releases]
enabled = true
draft = false
prerelease = false
generate_notes = false
source = "monochange"
```

### Recommendation on release notes

Prefer MonoChange-authored notes as the source of truth, not GitHub generated notes, because MonoChange has richer package/group semantics.

## Capability C — Release or changeset pull requests

### Why

Many repos want a branch and PR that contains the release commit, changelog updates, version bumps, and release notes before merging to `main`.

### Recommendation

Treat release PR automation as a first-class feature distinct from plain `mc release`.

### Desired behavior

A new automation flow should:

- read pending `.changeset/*.md`
- compute the release result
- apply file changes on a dedicated branch
- commit the release updates
- open or update a pull request
- keep one active release PR per configured lane or workflow

### Proposed config direction

```toml
[github.pull_requests]
enabled = true
branch_prefix = "monochange/release"
title = "chore(release): prepare release"
labels = ["release", "automated"]
auto_merge = false
```

### Important distinction

There are two PR concepts to support:

1. **changeset validation on feature PRs**
   - enforce whether a PR needs a changeset
2. **release PR creation**
   - aggregate pending changesets into a version-bump PR

MonoChange should model both explicitly.

## Capability D — Automatic deployments after merged release PRs

### Why

A release PR should often be the review checkpoint before deployment or publishing.

### Recommendation

MonoChange should provide structured metadata and workflow steps for deployment orchestration, but actual deploy commands should remain repository-configurable.

### Proposed model

After a release PR merges:

- CI detects a merged MonoChange release PR or a merge commit with release metadata
- MonoChange reconstructs or reads release metadata
- deployment steps run per configured environment or target
- deployment status is associated with the released version or tag

### Proposed config direction

```toml
[[deployments]]
name = "production"
trigger = "release_pr_merge"
workflow = "deploy-production"
requires = ["main"]
```

### Key recommendation

Do **not** hardcode platform-specific deployment behavior into MonoChange core yet. Instead:

- let MonoChange decide eligibility and metadata
- let repository workflows decide execution details

## Capability E — GitHub bot rules for changesets on PRs

### Why

Repositories need policy enforcement before merge:

- does this PR require a changeset?
- is the changeset valid?
- is a skip label allowed?
- do docs-only PRs bypass the requirement?
- are group/package references valid?

### Recommendation

Support a GitHub bot rules engine with an initial GitHub Action implementation and a future GitHub App path.

### Minimum rule set

- require a changeset when code affecting managed packages changes
- allow skip labels such as `no-changeset-required`
- validate that a referenced changeset is syntactically and semantically valid
- reject changesets that mix a group and one of its members
- optionally require evidence for certain package categories

### Proposed config direction

```toml
[github.bot.changesets]
enabled = true
required = true
skip_labels = ["no-changeset-required"]
comment_on_failure = true
changed_paths = ["crates/**", "packages/**"]
ignored_paths = ["docs/**", "*.md"]
```

## Recommended Architecture

## 1. Add a structured release-notes model

Introduce a core release-notes data structure that can drive:

- changelog file rendering
- GitHub release body rendering
- release PR body rendering

## 2. Add first-class automation steps to workflows

The current `Command` step is too generic for several GitHub-facing behaviors. Add typed steps such as:

- `PublishGitHubRelease`
- `OpenReleasePullRequest`
- `WriteReleaseManifest`
- `Deploy`
- `EnforceChangesetPolicy`

These steps can coexist with `Command` for repository-specific extensions.

## 3. Add a release manifest artifact

MonoChange should be able to emit a stable machine-readable artifact describing:

- release targets
- versions
- tags
- released packages
- changelog outputs, including structured note sections and rendered markdown
- the synchronized release plan that produced those outputs
- deployment intent

This manifest becomes the contract between:

- local CLI execution
- GitHub Actions
- future GitHub App automation

## 4. Keep GitHub API integration outside `monochange_core`

Recommended layering:

- `monochange_core` → pure release and workflow domain model
- `monochange_config` → parsing and validation of GitHub/bot config
- `monochange` → CLI orchestration and release preparation
- new crate, likely `monochange_github` → GitHub API client, PR/release operations, bot policy helpers

## 5. Start bot behavior in Actions before a full app

Phase 1 bot implementation can use:

- `pull_request` or `pull_request_target`
- `mc validate`
- a new `mc github check-pr` or `mc bot changeset-check`

If richer behavior is needed later, a GitHub App can reuse the same validation engine.

## Recommended Delivery Order

### Phase 1

- structured changelog formats
- shared release-notes rendering
- release manifest output

### Phase 2

- GitHub release publication
- release-note reuse for GitHub bodies

### Phase 3

- release PR creation and update
- branch/commit/PR templating

### Phase 4

- merge-driven deployment orchestration
- deployment metadata and status reporting

### Phase 5

- PR-time changeset policy bot
- skip labels, comments, and required-check behavior

## Key Risks

### Risk 1 — Scope explosion

This is five features, not one. Trying to ship them in one slice will likely overcouple the design.

### Risk 2 — GitHub semantics leaking into the core planner

The planner should stay reusable and mostly GitHub-agnostic.

### Risk 3 — Changelog and release-note divergence

Without a shared note model, file changelogs, PR bodies, and GitHub releases will drift.

### Risk 4 — Overcommitting on deploy providers too early

Deployment automation should start as orchestration plus metadata, not dozens of built-in providers.

### Risk 5 — Bot permissions and security

PR commenting and merge-aware automation require careful permission boundaries, especially for forks.

## Strong Recommendations

1. Ship changelog formats and release-note rendering before GitHub releases.
2. Ship GitHub releases before release PR automation.
3. Ship release PR automation before merge-driven deploy automation.
4. Start the changeset bot as an Actions-based policy check, then graduate to a GitHub App only if needed.
5. Introduce a dedicated `monochange_github` crate instead of placing GitHub HTTP logic directly in the CLI crate.
6. Add a machine-readable release manifest early because every later feature depends on it.

## Open Design Questions

1. Should release PRs always target `main`, or support configurable target branches?
2. Should GitHub releases be per package/group target, or support one aggregate release too?
3. Should changelog formats be independently configurable for package, group, PR body, and GitHub release body?
4. Should deployment automation happen inside MonoChange workflow steps, or primarily through emitted release manifests consumed by GitHub Actions?
5. Is a GitHub App an explicit near-term goal, or should the first bot release stay Actions-only?
